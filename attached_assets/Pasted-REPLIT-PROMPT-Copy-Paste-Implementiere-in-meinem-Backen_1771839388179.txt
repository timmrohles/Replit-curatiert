Implementiere in meinem Backend (Node/TypeScript bevorzugt, sonst Node/JS) ein internes Revenue-Share/Affiliate-Attributionssystem für Kuratoren mit folgender festen Logik:

Kontext

Unsere Plattform ist selbst Affiliate bei externen Buchhandlungen.

Kuratoren erhalten einen Anteil unserer tatsächlich erhaltenen Affiliate-Provision.

Es gibt zwei potenzielle Attributionsquellen:

Extern angeworbener Kunde (User kommt erstmals über einen personalisierten Kuratoren-Link)

Kuration-Klick (User klickt ein Buch innerhalb einer Kuration eines Kurators)

WICHTIGE PRIORITÄTSREGEL (exklusiv)

Wenn ein User über einen Kuratoren-Link (extern angeworben) auf die Plattform gekommen ist und innerhalb des Attributionsfensters kauft, erhält ausschließlich dieser Kurator die Vergütung.

In diesem Fall darf keine Kurationsvergütung (auch nicht an andere Kuratoren) für denselben Kauf entstehen.

Nur wenn kein extern angeworbener Kurator zugeordnet ist, darf eine Kurationsvergütung greifen.

Anforderungen (Technik)

Tracking beim Entry

Wenn ein Besucher über einen Kuratoren-Link kommt (z. B. /@{creatorSlug} oder ?ref={creatorId}), setze:

serverseitig einen Cookie oder Session-Key ref_creator_id

speichere einen DB-Record referral_sessions mit:

session_id

ref_creator_id

first_seen_at

expires_at (Attributionsfenster, z. B. 7 Tage)

landing_url

Tracking beim Kurationsklick

Wenn ein User auf ein Buch innerhalb einer Kuration klickt, logge:

curation_clicks:

session_id

curation_id

curation_owner_creator_id

book_id (oder isbn)

clicked_at

Attribution beim Kauf (Conversion)

Wir bekommen später eine Conversion/Commission (z. B. via Import/Callback). Implementiere eine Funktion:

attributeConversion(conversion)

Input conversion enthält mindestens:

session_id (oder click_id)

book_id/isbn

merchant

commission_amount_net

occurred_at

external_order_id (optional)

Logik:

Prüfe zuerst, ob eine gültige referral_sessions Zuordnung existiert (nicht expired).

Wenn ja: setze attributed_creator_id = ref_creator_id und attribution_type = 'REFERRAL'

Wenn nein: finde den letzten passenden curation_click für diese session + book (innerhalb z. B. 24h)

Wenn gefunden: attributed_creator_id = curation_owner_creator_id, attribution_type = 'CURATION'

Sonst: keine Attribution

Schreibe Ergebnis in Tabelle creator_commissions (idempotent per unique key auf external_order_id oder (merchant, occurred_at, session_id, book_id))

Provision Calculation

Creator-Payout = commission_amount_net * revenue_share_rate

revenue_share_rate ist konfigurierbar (z. B. 0.5)

Speichere:

gross_commission_net

share_rate

creator_payout_amount

status: pending/confirmed/cancelled

Idempotenz & Storno

Dedupe: gleiche Conversion darf nicht doppelt verbucht werden.

Stornos/Chargebacks müssen creator_commissions auf cancelled setzen und Payout rückrechnen.

DB (Neon Postgres)

Erstelle Migrationen/SQL (oder Prisma/Drizzle Models) für Tabellen:

referral_sessions(session_id PK, ref_creator_id, first_seen_at, expires_at, landing_url)

curation_clicks(id PK, session_id, curation_id, curation_owner_creator_id, book_id, clicked_at)

creator_commissions(id PK, external_order_id UNIQUE NULL, merchant, session_id, book_id, attribution_type, attributed_creator_id, commission_amount_net, share_rate, creator_payout_amount, occurred_at, status)

API / Endpoints

GET /r/:creatorSlug oder Middleware, die Referral setzt und weiterleitet

POST /track/curation-click

POST /track/conversion (für Import/Callback)
Alle Endpoints müssen serverseitig validieren und loggen.

Output
